# 多路温湿度采集监控系统

## 配置文件

### 基础配置文件

基础配置文件有三个，分别是Device.ini（通信参数配置文件）、Group.xlsx（通信组配置文件）和Variable.xlsx（通信变量配置文件）。三个基础配置文件只要有一个为空，就必须配置完毕后重启系统才能生效。



#### 通信参数配置文件

Device.ini（通信参数配置文件）的内容包括：IP地址、端口号和当前配方名（可选）。路径为：`Application.StartupPath + "\\Config\\Device.ini"`。

#### 通信组配置文件

在本项目中，一次设备通信数据读取的单位为组，一次传输就将一组内的数据全部读取出来。通信组实体类的成员内容如下：

| 通信组实体类成员 | 含义                     |
| ---------------- | ------------------------ |
| `GroupName`      | 通信组名称               |
| `StartAddress`   | 该通信组的起始硬件地址   |
| `Length`         | 该通信组包含的硬件个数   |
| `StoreArea`      | 该通信组所在的存储区名称 |
| `Remark`         | 备注说明                 |

通信组配置文件路径为：`Application.StartupPath + "\\Config\\Group.xlsx"`。

#### 通信变量配置文件

在本项目中，对于读取到的每组数据，按照通信变量配置文件所记录的通信变量解析成多个有意义的数据。通信变量实体类的成员内容如下：

| 通信变量实体类成员 | 含义                                                         |
| ------------------ | ------------------------------------------------------------ |
| `VarName`          | 通信变量名称                                                 |
| `Start`            | 变量在寄存器中存储的情况下表示组内变量起始字索引；在线圈中存储的情况下表示组内变量起始字节索引 |
| `DataType`         | 变量的数据类型                                               |
| `OffsetOrLength`   | 变量类型是布尔，表示字节内位偏移；类型是字符串或字节数组，表示长度 |
| `GroupName`        | 所属通信组的名称                                             |
| `PosAlarm`         | 该变量是否需要上升沿报警                                     |
| `NegAlarm`         | 该变量是否需要下降沿报警                                     |
| `Scale`            | 转换系数，用于变量值进行线性转换                             |
| `Offset`           | 偏移量，用于变量值进行线性转换                               |
| `Remark`           | 备注说明                                                     |

通信变量配置文件路径为：`Application.StartupPath + "\\Config\\Variable.xlsx"`。

### 配方配置文件

配方配置文件使用`.ini`文件，文件名即为配方名。配方数据包含配方名以及各个站点的温湿度高低限和温湿度报警使能，其中各个站点的温湿度高低限和温湿度报警使能数据是以`JSON`格式保存在配置文件中的。配方配置文件所在的文件夹路径为：`Application.StartupPath + "\\Recipe"`。

## 数据库

本项目选用SQL Server数据库，程序使用ADO.NET技术以访问它。共创建了3个数据表，分别是用户表、报警日志表和历史数据表。

### 用户数据表

| 字段名         | 字段类型     | 字段含义                 |
| -------------- | ------------ | ------------------------ |
| `LoginId`      | int          | 用户ID（主键、自增）     |
| `LoginName`    | varchar(100) | 用户名（唯一键）         |
| `LoginPwd`     | varchar(20)  | 用户密码                 |
| `ParamSet`     | bit          | 用户访问“参数设置”的权限 |
| `Recipe`       | bit          | 用户访问“配方管理”的权限 |
| `HistoryLog`   | bit          | 用户访问“报警追溯”的权限 |
| `HistoryTrend` | bit          | 用户访问“历史趋势”的权限 |
| `UserManage`   | bit          | 用户访问“用户管理”的权限 |

### 报警日志数据表

| 字段名       | 字段类型     | 字段含义                     |
| ------------ | ------------ | ---------------------------- |
| `Id`         | int          | 日志ID（主键、自增）         |
| `InsertTime` | varchar(50)  | 日志插入时间                 |
| `Note`       | varchar(200) | 日志内容                     |
| `AlarmType`  | varchar(20)  | 报警类型，报警触发或报警消除 |
| `Operator`   | varchar(100) | 操作员，即当前用户           |
| `VarName`    | varchar(20)  | 报警的变量名称               |

### 历史数据表

历史数据表的字段包含Id、数据记录时间以及各站点当时的温湿度测量数据。

## 项目架构

- 项目总体分成集中监控、参数设置、配方管理、报警追溯、历史趋势和用户管理6大模块，功能丰富；各模块内聚性强，易于维护。
- 整体地，本项目的UI渲染和设备通信（控制器数据读取）分属不同的线程，既提高了程序性能，又降低了UI渲染卡顿的风险。
- 在项目涉及到数据库访问的模块，分成了UI层、业务层（BLL）和数据访问层（DAL），架构清晰了然。
- 项目维护有一个全局静态变量键值对，通信线程读取数据并解析后会更新该全局键值对存储的值，UI线程读取该键值对存储的值用于更新界面显示。（[C#规范](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/language-specification/variables#atomicity-of-variable-references)明确指出，对 `bool`、`char`、`int` 等不大于 32 位的值类型的读写是原子的，故本项目在这里暂不考虑对单个变量访问的加锁问题；而如果对整个`Dictionary`加锁，系统的性能会明显下降，故这里不加锁，牺牲了部分数据更新的统一性）
- 项目还包含了自己实现的`ModbusTCP`通信库和自定义控件库。自实现的`ModbusTCP`通信库包含了线圈与寄存器的读取，单个和多个线圈或寄存器的写入功能；实现遵循标准Modbus协议并且充分考虑了线程安全和异常安全。自定义控件库则通过组合现有控件和利用GDI+绘制技术实现了许多必要的组件和显示控件，包括仪表盘控件、增强版Panel组件和各种组合控件等。
